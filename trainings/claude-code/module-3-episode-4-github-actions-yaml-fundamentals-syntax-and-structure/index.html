üîë Core Concepts Explained

Concept 1: YAML Structure

Definition:
YAML is a human-readable data format using indentation for nesting, hyphens for lists, and colons for key-value pairs.

Why It Matters:
Incorrect indentation breaks workflows. Understanding structure prevents silent failures.

When To Use:
Always; it's the foundation of GitHub Actions.

Concept 2: Workflow Triggers

Definition:
Events that cause workflows to execute (push, pull_request, schedule, manual trigger).

Why It Matters:
Different tasks need different triggers. Wrong trigger = automation doesn't run when needed.

When To Use:
For every workflow‚Äîbe explicit about when it runs.

Concept 3: Job & Step Structure

Definition:
Workflows contain jobs; jobs contain steps. Each step is an action or shell command.

Why It Matters:
Understanding structure lets you compose complex workflows from simple pieces.

When To Use:
In every workflow.

‚öôÔ∏è Technical Deep Dive

YAML Basics

# Basic structure
name: Workflow Name
on: push
jobs:
  job_id:
    runs-on: ubuntu-latest
    steps:
      - uses: action/name@version
        with:
          param: value
      - run: shell command
        env:
          VAR: value

Triggers

on:
  # Trigger on push to main
  push:
    branches: [main]

  # Trigger on pull request
  pull_request:
    branches: [main]

  # Trigger on schedule (cron)
  schedule:
    - cron: '0 0 * * *'  # Daily at midnight

  # Manual trigger
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deployment environment'
        required: true
        default: 'staging'

Matrix Strategy (Run Multiple Configurations)

jobs:
  test:
    strategy:
      matrix:
        node-version: [14, 16, 18]
        os: [ubuntu-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/setup-node@v3
        with:
          node-version: ${{ matrix.node-version }}

Environment Variables & Secrets

env:
  GLOBAL_VAR: 'visible to all steps'

jobs:
  build:
    env:
      JOB_VAR: 'visible to all steps in this job'
    steps:
      - run: echo $JOB_VAR
        env:
          STEP_VAR: 'visible only to this step'
          SECRET_VALUE: ${{ secrets.API_KEY }}

Conditional Execution

steps:
  - name: Run on success
    if: success()
    run: echo "All previous steps succeeded"

  - name: Run on failure
    if: failure()
    run: echo "A previous step failed"

  - name: Run on specific condition
    if: github.ref == 'refs/heads/main'
    run: echo "Running on main branch"

Outputs & Artifacts

jobs:
  build:
    outputs:
      build-number: ${{ steps.build.outputs.number }}
    steps:
      - id: build
        run: echo "number=$(date +%s)" >> $GITHUB_OUTPUT

  deploy:
    needs: build
    steps:
      - run: echo "Built version ${{ needs.build.outputs.build-number }}"

  test:
    needs: build
    steps:
      - uses: actions/download-artifact@v3
        with:
          name: build-output

‚úÖ Common YAML Patterns

Pattern 1: Dependency Management

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci && npm run lint

  test:
    needs: lint  # Wait for lint to succeed
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci && npm test

  build:
    needs: test  # Wait for test to succeed
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci && npm run build

Pattern 2: Conditional Deployment

  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # Only on main branch
    steps:
      - uses: actions/checkout@v3
      - name: Deploy to production
        run: npm run deploy -- --env=production

Pattern 3: Reusable Workflow

# reusable-workflow.yml
name: Reusable Test Workflow

on:
  workflow_call:
    inputs:
      node-version:
        required: true
        type: string

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm test

# main-workflow.yml
jobs:
  test-matrix:
    uses: ./.github/workflows/reusable-workflow.yml
    with:
      node-version: '18'

üí° Practical Tips & Tricks

Pro Tip 1: Debug Workflow Runs

The Technique: Enable debug logging to see all variables.

env:
  ACTIONS_STEP_DEBUG: true

Expected Result: Detailed logs show variable substitution and step execution.

Pro Tip 2: Validate YAML Before Committing

The Technique: Use yamllint locally.

pip install yamllint
yamllint .github/workflows/

Expected Result: Catch YAML errors before pushing.

Pro Tip 3: Reuse with Composite Actions

The Technique: Create reusable action wrapper.

# composite-action.yml
name: Setup and Test

runs:
  using: composite
  steps:
    - uses: actions/setup-node@v3
      with:
        node-version: 18

    - run: npm ci
      shell: bash

    - run: npm test
      shell: bash

üõ† Implementation Guide

Step 1: Create Workflow File

mkdir -p .github/workflows
cat > .github/workflows/ci.yml << 'EOF'
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '18'
      - run: npm ci
      - run: npm test
EOF

Step 2: Commit and Push

git add .github/workflows/ci.yml
git commit -m "Add CI workflow"
git push origin main

ü§î Common Questions & Troubleshooting

Q: Why is my workflow not running?

A: Check triggers. Use workflow_dispatch for manual testing.

Q: How do I use secrets in workflows?

A: Define in repository settings; access via ${{ secrets.NAME }}.

Q: Can I test workflows locally?

A: Yes, use act tool: brew install act && act push.
